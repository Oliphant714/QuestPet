## 1. Problem Statement

The Add Task and Edit Task popups in `TaskWindow` work but have two bugs: the difficulty
dropdown defaults to `"Easy"` while `Task.__init__` defaults to `"medium"`, so saving
without choosing a difficulty sets a mismatched value. Additionally the popup geometry
is too small (200px height) to comfortably show all five fields. These gaps create a
confusing first-use experience and can produce incorrect XP rewards.

## 2. Goals

- The Add Task popup default difficulty matches `Task`'s default (`"medium"`).
- All five fields (title, description, difficulty, due date) are comfortably visible
  without scrolling.
- Due-date parse errors are shown inline below the field rather than in a blocking
  `messagebox`.
- Saving with a blank title is prevented with a clear inline message.
- Changes are persisted via `TaskManager.save_state` and reflected immediately in the
  listbox.

## 3. Non-Goals

- Rich text descriptions or markdown rendering.
- Recurring tasks or subtasks.
- Drag-and-drop reordering of tasks.

## 4. Personas & User Stories

- **As a player**, I want to add a new quest with a title, optional description,
  difficulty, and optional due date so that I can track what I need to do.
- **As a player**, I want to edit a task's fields after creating it so that I can
  correct mistakes or update a deadline.
- **As a player**, I want to see an error message next to the field that has a problem
  instead of a popup dialog so that I know exactly what to fix.

## 5. Functional Requirements

- **FR-1.** The system MUST default the difficulty dropdown in Add Task to `"Medium"`
  (matching `Task`'s `"medium"` default, display-capitalised).
- **FR-2.** The system MUST prevent saving a task with an empty title and display an
  inline label `"Title is required"` in red below the title field.
- **FR-3.** The system MUST display an inline error `"Use YYYY-MM-DD format"` below the
  due-date field if the date string cannot be parsed, without closing the popup.
- **FR-4.** The system MUST call `task_manager.save_state()` after every successful
  add or edit.
- **FR-5.** The system MUST call `self.refresh()` after every successful add or edit so
  the listbox reflects the change immediately.
- **FR-6.** The system SHOULD resize the Add/Edit popup to at least 300×280 px to
  accommodate all fields without clipping.

## 6. Non-Functional Requirements

- **Performance** — Popup opens in < 200 ms.
- **Maintainability** — Add and Edit share the same popup builder function with an
  optional `task=None` argument; no duplicated widget creation code.

## 7. Acceptance Criteria

- **AC-1.** *Given* the Add Task popup is opened, *when* the difficulty dropdown is
  inspected before the user touches it, *then* it shows `"Medium"`.
- **AC-2.** *Given* the Add Task popup is open with title blank, *when* Save is clicked,
  *then* the popup remains open and a red `"Title is required"` label appears; no task
  is created.
- **AC-3.** *Given* the Add Task popup is open with due date `"not-a-date"`, *when*
  Save is clicked, *then* an inline error appears below the due-date field and no task
  is created.
- **AC-4.** *Given* a valid title and difficulty are entered, *when* Save is clicked,
  *then* the popup closes, the new task appears in the active listbox, and the save file
  is updated.
- **AC-5.** *Given* an existing task is selected and Edit is clicked, *when* the popup
  opens, *then* all existing field values are pre-populated correctly.

## 10. UI / UX

- Popup size: `300x280`.
- Error labels: `fg="red"`, placed directly below the offending field, hidden until
  an error occurs.
- Difficulty options: `"Easy"`, `"Medium"` (default), `"Hard"`.
- Save button: disabled (greyed) while title field is empty, re-enabled on keystroke.
- Flow: open popup → fill fields → click Save → popup closes → listbox refreshes.
- Error flow: click Save with bad input → popup stays open → inline error visible →
  user corrects → Save succeeds.
