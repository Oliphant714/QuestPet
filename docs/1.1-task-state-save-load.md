## 1. Problem Statement

When QuestPet closes and reopens, active and completed tasks are reloaded from
`questpet_tasks.json`, but `Task.next_id` can drift if the JSON and the class-level
counter fall out of sync, causing duplicate IDs. Additionally, `PetCore` state
(level, XP, growth points) is never written to disk, so the pet resets to level 1
every session. Users lose all progression on every restart, undermining the core
gamification loop.

## 2. Goals

- Active and completed tasks survive app restart with all fields intact.
- `Task.next_id` is always reconstructed correctly from persisted data.
- `PetCore` level, XP, and growth points are saved and restored each session.
- Save is atomic ‚Äî a crash during write does not corrupt the existing save file.
- Load gracefully handles a missing or malformed save file (first-run scenario).

## 3. Non-Goals

- Cloud sync or multi-device support.
- Save file versioning / migration for schema changes (covered by story 1.2).
- Encrypting the save file.

## 4. Personas & User Stories

- **As a player**, I want my tasks to still be there when I reopen QuestPet so that I
  don't have to re-enter them every session.
- **As a player**, I want my pet's level and XP to persist so that my progression feels
  meaningful over time.
- **As a developer**, I want save/load to be robust so that a crash or power loss does
  not leave users with a blank slate.

## 5. Functional Requirements

- **FR-1.** The system MUST write all active tasks, completed tasks, streak, last completion
  date, `Task.next_id`, pet level, pet XP, and pet growth points to
  `questpet_tasks.json` on every `save_state` call.
- **FR-2.** The system MUST write to a `.tmp` file first and rename atomically so a
  mid-write crash does not corrupt the existing save.
- **FR-3.** The system MUST reconstruct `Task.next_id` as `max(saved next_id, max(task.id) + 1)`
  on load, whichever is larger.
- **FR-4.** The system MUST load `PetCore` fields (level, xp, xp_to_next, growth_points)
  from the save file when `load_state` is called.
- **FR-5.** The system SHOULD log a warning and start fresh if the save file is present
  but cannot be parsed as valid JSON.
- **FR-6.** The system MAY keep a single `.bak` copy of the previous save before
  overwriting.

## 6. Non-Functional Requirements

- **Performance** ‚Äî Save completes in < 100 ms for up to 500 tasks.
- **Security** ‚Äî Save file is plain JSON on the local filesystem; no network exposure.
- **Privacy** ‚Äî No PII beyond what the user types into task fields.
- **Reliability** ‚Äî Atomic write (tmp + rename) prevents partial-write corruption.
- **Maintainability** ‚Äî Save/load logic confined to `TaskManager.save_state` /
  `load_state`; `PetCore` exposes `to_dict` / `from_dict` mirroring `Task`.

## 7. Acceptance Criteria

- **AC-1.** *Given* a session with two active tasks and pet at level 2, *when* the app
  closes and reopens, *then* both tasks appear in the active list and the pet is still
  level 2 with the same XP.
- **AC-2.** *Given* a valid save file, *when* `questpet_tasks.json` is deleted before
  launch, *then* the app starts with an empty task list and a level-1 pet without
  crashing.
- **AC-3.** *Given* a save file with `next_task_id: 5` but tasks with IDs up to 7,
  *when* the app loads, *then* the next new task receives ID 8.
- **AC-4.** *Given* a save file containing invalid JSON, *when* the app loads, *then*
  a warning is printed and the app starts fresh without raising an unhandled exception.

## 8. Data Model

The existing JSON structure in `questpet_tasks.json` gains a `pet_core` key:

```json
{
  "streak": 3,
  "last_completion_date": "2026-06-25",
  "next_task_id": 12,
  "pet_core": {
    "level": 2,
    "xp": 45,
    "xp_to_next": 150,
    "growth_points": 1
  },
  "active_tasks": [...],
  "completed_tasks": [...]
}
```

`PetCore` gains `to_dict()` and `from_dict(cls, data)` class methods mirroring the
pattern already used by `Task`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Atomic rename fails on Windows (cross-device move) | L | H | Use `pathlib.Path.replace()` which is atomic on Windows NTFS within the same drive |
| `PetCore` reference not wired into `TaskManager` in `main.py` | M | M | Update `main.py` in the same PR; add an assertion in `load_state` |
| Existing save files missing `pet_core` key break load | M | H | Default to level-1 values if key absent (backward-compatible load) |
